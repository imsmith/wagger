# Wagger MCP Provider — Design

**Date:** 2026-04-17
**Status:** Approved for planning
**Scope:** First slice of MCP support in wagger. Emits `my-app-mcp.yang` source text from a flat capability map. Contexter runtime and `mcp2yang` ingest parser are separate specs.

## Context

Wagger today is a WAF rule generator: one `Wagger.Generator` behaviour, one provider module per target (aws, caddy, coraza, etc.), YANG sources in `/yang/wagger-<provider>.yang`. Each provider maps HTTP routes to a native config format.

MCP (Model Context Protocol) support fits the same generator pattern but projects a different input domain — app *capabilities* (tools, resources, prompts) rather than HTTP routes — to a different output — a YANG module that imports a canonical `mcp.yang`.

Three projects form the eventual MCP round-trip: **wagger** (generator, this spec), **contexter** (runtime framework, separate project), **mcp2yang** (ingest parser, separate project). This spec is wagger's piece only.

## Goals

1. Wagger emits well-formed `my-app-mcp.yang` source text from a capability map.
2. The emitted module imports a canonical `mcp.yang` shipped with wagger.
3. Existing WAF provider pipeline is untouched.
4. Struct-based emission: provider builds `ExYang.Model.Module{}`, encoder produces text. Round-trip parse+resolve validates output.

## Non-goals

- No contexter runtime, no Elixir codegen for MCP server adapters.
- No Phoenix UI surface for the MCP generator.
- No YANG-annotation conventions for deriving capability maps from existing YANG modules (`mcp2yang`'s future inverse).
- No concrete typed refinement of per-tool JSON Schema — tools' `input_schema`/`output_schema` emit as `anydata` with the JSON Schema attached as a `description` annotation.
- No semantic validation against the MCP JSON-RPC spec — that's contexter's job.

## Architecture

Two new artifacts, one behaviour change:

- `yang/mcp.yang` — canonical base module, revision `2025-06-18`. Hand-authored, version-controlled.
- `lib/wagger/generator/mcp.ex` — provider module. Implements capability-shaped callback.
- `lib/wagger/generator/mcp/builder.ex` — pure functions, capability map → `ExYang.Model.*` structs. Split from the provider so tests can assert on struct shape without encoder round-trip.
- `Wagger.Generator` behaviour — gains one optional callback `map_capabilities/2`.

### Behaviour change

```elixir
@callback map_capabilities(capabilities :: map(), config :: map()) :: ExYang.Model.Module.t()
```

`Wagger.Generator.generate/3` dispatches on which callback the provider exports:

- If `function_exported?(provider, :map_capabilities, 2)` — capability pipeline.
- Else — existing route pipeline, unchanged.

WAF providers keep implementing `map_routes/2` and `serialize/2`. MCP provider implements `yang_module/0` and `map_capabilities/2` only.

## Canonical `mcp.yang` scope

Full protocol surface, revision `2025-06-18`:

- **Groupings** — `tool-definition`, `resource-definition`, `prompt-definition`, `capabilities`, `server-info`.
- **Lifecycle RPCs** — `initialize`, `ping`, `tools/list`, `tools/call`, `resources/list`, `resources/read`, `prompts/list`, `prompts/get`.
- **Notifications** — `tools/list_changed`, `resources/updated`, `prompts/list_changed`.
- **Transport identities** — `stdio`, `streamable-http`, `sse`.

Dynamic per-tool JSON Schema in `tool-definition` is modeled as `anydata input-schema` / `anydata output-schema`. Future spec can refine to concrete types via `deviation`.

## Data flow

Input:

```elixir
capabilities = %{
  app_name: "my-app",  # required, must match YANG identifier rules
  tools: [
    %{name: "search", description: "...",
      input_schema: %{...}, output_schema: %{...}}
  ],
  resources: [
    %{uri_template: "file://{path}", name: "file", mime_type: "text/plain"}
  ],
  prompts: [
    %{name: "summarize", arguments: [%{name: "length", required: true}]}
  ]
}
config = %{}  # reserved; unused in v1
```

Pipeline for capability providers:

1. Orchestrator detects `map_capabilities/2` is exported; takes capability branch.
2. `provider.yang_module/0` returns canonical `mcp.yang` source (read from `yang/mcp.yang`, or baked in via `@external_resource`).
3. Canonical source is parsed+resolved via `ExYang.parse/1` + `ExYang.resolve/2`. Cheap self-check.
4. `Builder.validate/1` checks capability map; returns `{:error, ...}` on malformed input before struct construction.
5. `provider.map_capabilities/2` constructs an `ExYang.Model.Module{}` with:
   - `name = "#{app_name}-mcp"`, `namespace = "urn:wagger:#{app_name}:mcp"`, `prefix = app_name` (derived — convention over configuration).
   - `Import{module: "mcp", prefix: "mcp", revision_date: "2025-06-18"}`.
   - One `Container` per primitive holding `Uses{name: "mcp:tool-definition"}` (etc.) projections.
   - `Revision{date: <build date>, description: "generated by wagger"}`.
6. `ExYang.Encoder.Encoder.encode/1` → YANG source string.
7. Round-trip validation: re-parse the emitted string, resolve against a registry containing the canonical `mcp.yang`. Failure → `:internal` error (wagger bug).
8. Return `{:ok, yang_source}`.

## Module identity

Convention: caller provides `app_name`, wagger derives the rest.

- `module_name = "#{app_name}-mcp"`
- `namespace = "urn:wagger:#{app_name}:mcp"`
- `prefix = app_name`

Override via explicit fields in the capability map is not supported in v1. Add later if a caller needs a registered IANA namespace.

## Error handling

Three error IDs under `wagger.generator/`:

- **`invalid_capabilities`** (`:validation`) — malformed input. Missing `app_name`, invalid identifier, missing required primitive field, duplicate names within a primitive list. `field:` points to the offending path (`"tools[2].name"`). Caught by `Builder.validate/1`.
- **`mcp_roundtrip_failed`** (`:internal`) — encoder emitted text that fails to re-parse or re-resolve. Wagger bug; error message includes the ex_yang diagnostic.
- **`canonical_mcp_invalid`** (`:internal`) — canonical `mcp.yang` itself fails to parse/resolve. Fatal; the shipped file is broken.

Validation happens once up front in `Builder.validate/1`, not scattered across struct helpers.

## Components

| Unit | Path | Role |
|---|---|---|
| Canonical mcp.yang | `yang/mcp.yang` | Hand-authored base module, version-controlled |
| Provider | `lib/wagger/generator/mcp.ex` | Implements `yang_module/0` + `map_capabilities/2` |
| Builder | `lib/wagger/generator/mcp/builder.ex` | Pure: capability map → `ExYang.Model.*` structs; validation |
| Behaviour | `lib/wagger/generator/generator.ex` | Adds optional `map_capabilities/2` + dispatch in `generate/3` |

No new deps. No Phoenix/UI changes. No contexter.

## Testing

**`test/wagger/generator/mcp/builder_test.exs`** — pure struct assertions. Minimal capability map, one of each primitive, many of each, every validation error path. No encoder, no parser.

**`test/wagger/generator/mcp_test.exs`** — full pipeline via `Wagger.Generator.generate/3`. Assertions: returns `{:ok, source}`, source contains expected module/namespace/revision, source round-trips, resolves against canonical `mcp.yang`. One golden-file fixture compared byte-for-byte; regenerate flow documented in a module comment.

**`test/wagger/generator/mcp/canonical_test.exs`** — `yang/mcp.yang` parses, resolves against empty registry, all declared groupings/RPCs/notifications are reachable, revision date equals `2025-06-18`.

Deliberately out of scope: MCP spec semantic conformance, running-server behavior.

## Open questions for implementation

- Load canonical `mcp.yang` at compile time (`@external_resource` + module attribute) or runtime (`File.read!` at first call with caching)? Decide during implementation; both are fine.
- Where exactly does the primitive-to-YANG projection place concrete tool names — as children under a `tools` container with `Uses{name: "mcp:tool-definition"}`, or as augments to the canonical `tools` container? Resolve during builder implementation against what `ExYang.Model` allows cleanly.

## Follow-on specs (not this one)

- `mcp2yang` ingest parser (separate project).
- Contexter runtime framework (separate project).
- YANG-annotation conventions so `map_capabilities/2` can be derived from a YANG module's `rpc` and `container` nodes.
- Concrete typed refinement of tool `input_schema`/`output_schema` via `deviation`.
- Phoenix UI for the MCP generator alongside WAF providers.
